US-24: Xác Thực
User Story
	Là người dùng, tôi muốn access token được tự động làm mới khi hết hạn mà không cần đăng nhập lại, để trải nghiệm sử dụng liên tục và bảo mật.
	Acceptance Criteria
	* Khi đăng nhập thành công, server set HttpOnly cookie chứa refresh token (Secure; SameSite=Strict; Path=/api/v1/auth/refresh); accessToken trả về trong body response.
* Khi accessToken hết hạn (401), frontend tự động gọi POST /api/v1/auth/refresh — trình duyệt tự đính cookie, không cần thêm code gửi token.
* Server thực hiện token rotation: revoke refresh token cũ trong DB, set cookie mới, trả accessToken mới trong body.
* Nếu refresh token cũ đã bị rotation được dùng lại (reuse attack) → server revoke toàn bộ session của user, trả 401.
* Refresh token không accessible qua JavaScript (HttpOnly) — kiểm tra bằng browser devtools.
* Khi đăng xuất, server revoke refresh token trong DB và xóa cookie (Set-Cookie: refreshToken=; Max-Age=0); frontend xóa accessToken khỏi memory.
* Refresh token hết hạn (cookie expired hoặc bị revoke) → /auth/refresh trả 401 → frontend redirect về trang đăng nhập.

US-06: Quản Lý Sản Phẩm
User Story
	Là Admin, tôi muốn xóa sản phẩm không còn sử dụng để giữ danh mục gọn gàng.
	Acceptance Criteria
	* Nhấn 'Xóa' → hệ thống kiểm tra sản phẩm có giao dịch nhập/xuất chưa.
* Chưa có giao dịch → hiện hộp xác nhận → xác nhận → hard delete khỏi DB, sản phẩm biến hoàn toàn khỏi hệ thống.
* Đã có giao dịch → hiện thông báo giải thích → xác nhận → soft delete (is_deleted = true): ẩn khỏi danh sách, không thể chọn khi tạo phiếu mới, lịch sử giao dịch vẫn giữ nguyên.
* Sản phẩm có tồn kho > 0 khi soft delete → hiện cảnh báo bổ sung yêu cầu xác nhận lần hai.
